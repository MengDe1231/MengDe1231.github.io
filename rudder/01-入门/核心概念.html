核心概念 ​

Rudder 解决什么问题 ​

用过 AI 写代码的都懂——每次新会话打开，AI 就像喝了孟婆汤，一切从头来。

Rudder 做的事很简单：把项目规范、记忆、任务状态全部落盘到仓库里，每次会话自动按需注入，AI 不再"失忆"。

分层架构 ​

Rudder 不把 3000 行规则塞进一个文件，而是分层：

层级位置作用
工作流定义.rudder/workflow.md4 阶段循环的总纲
Spec 规范.rudder/spec/按作用域拆分的编码指南
任务上下文.rudder/tasks/每个任务的 PRD、实现记录、审查上下文
项目记忆.rudder/workspace/开发者工作日志，跨 session 续上下文
平台适配.claude/ .cursor/ .codex/ ...各平台自动触发的 hooks 和 agents

为什么分层 ​

  1. CLAUDE.md / AGENTS.md / .cursorrules 的痛点：用久了变成几千行的缝合怪，没人读
  2. Spec 按作用域拆分：前端、后端、数据库各有不同规范，不需要塞一个文件
  3. 任务独立上下文：每个任务有自己的 PRD、实现记录和审查结论
  4. 工作流有关卡：Plan → Implement → Verify → Finish，每个阶段有明确要求
  5. 跨平台自动适配：同一套 Rudder 结构，随便切 Claude Code / Gemini / Cursor / Codex

核心原则 ​

  1. 规范写好一次 — 放 .rudder/spec/ 里，Rudder 按任务自动注入相关文件
  2. 一切落盘 — 研究、决策、教训全写文件，对话会被压缩，文件不会
  3. 增量开发 — 一个任务一个任务来
  4. 沉淀认知 — 每个任务结束后，把新认知写回 Spec

谁在用 ​

  • 个人开发者：项目记忆 + 可复用工作流
  • 团队：标准统一、任务边界清晰、上下文可审查、换平台不换脑子
  • Java 全栈工程师：Rudder 作者本人是 Java 全栈开发，日常自用迭代，Java spec 模板最丰富；同时其他语言也完全兼容
