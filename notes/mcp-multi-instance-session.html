MCP Server 多实例 Session 丢失问题

解决方案

On this page

MCP Server 多实例 Session 丢失问题 — 解决方案 ​

1. 问题描述 ​

MCP Server 采用 SSE（Server-Sent Events）长连接协议，session 存储在应用内存中（WebMvcSseServerTransportProvider 内部 ConcurrentHashMap<String, McpServerSession>）。

多实例部署时，SSE 连接建立与后续 POST 请求被 WAF/SLB 轮询分发到不同实例，目标实例内存中不存在该 session，返回 "Session not found" 错误。

问题链路 ​

客户端 → GET /server/sse → WAF(轮询) → 实例A
         ├── 实例A 创建 session（id=744d...）
         └── 返回 SSE 事件流：event: endpoint, data: /mcp/message?sessionId=744d...

客户端 → POST /server/mcp/message?sessionId=744d... → WAF(轮询) → 实例B
         ├── 实例B 内存中无 session 744d...
         └── 返回: "Session not found: 744d..."  ← 404

约束条件 ​

  • 阿里云 WAF 只支持 IP Hash，不支持基于 URL 参数（sessionId）的一致性哈希
  • IP Hash 会导致同一出口 IP 的流量集中到单实例，失去多实例负载均衡意义
  • MCP SDK 的 McpServerSession 包含 SseEmitter、请求队列等不可序列化对象，无法直接存入 Redis

3. 方案 A：切换为 STATELESS 协议 ​

原理 ​

MCP SDK 0.17.0 提供三种传输模式：

模式session 状态端点多实例安全
SSE有状态GET /sse + POST /mcp/message❌
STREAMABLE有状态POST /mcp❌
STATELESS无状态POST /mcp✅

STATELESS 模式下，服务端不维护任何 session 状态，每次请求独立处理。客户端每次请求都会携带完整上下文，服务端无需记忆历史交互。

Nacos 配置 ​

yaml
spring:
  ai:
    mcp:
      server:
        protocol: STATELESS
        enabled: true
        name: chancemate-ai-mcp-server
        version: 1.0.0
        streamable-http:
          mcp-endpoint: /mcp

4. 方案 B：SSE + 应用层 Session 路由转发（当前采用） ​

原理 ​

保持 SSE 协议不变，客户端完全无感知。应用层通过以下机制解决跨实例问题：

  1. SSE 连接建立时，即时捕获 sessionId 并注册到 Redis（记录 sessionId → 实例地址映射）
  2. POST 请求到达时，查 Redis 获取 session 归属实例
  3. 如果不是本实例，HTTP 转发到正确实例并将响应原样返回

   → 实例B 将响应原样返回给客户端

双层 Session 注册机制 ​

层次触发时机实现方式作用
即时注册（主）SSE 连接建立时包装 ServletOutputStream，实时正则匹配消除时间窗口，确保 initialize 不 404
定时兜底（辅）每 2 秒反射扫描 SDK 内存 session map补偿即时注册可能遗漏的极端场景

为什么需要两层？

SSE 连接建立后，客户端会立即（通常 <100ms）发送 initialize POST 请求。如果只依赖定时扫描器（2 秒间隔），此时 sessionId 尚未同步到 Redis，跨实例路由失败 → "Session not found"。即时注册彻底消除了这个时间窗口。

新增代码详解 ​

McpSessionRegistry.java — Redis 注册服务 ​

java
@Service
@ConditionalOnProperty(name = "mcp.cluster.enabled", havingValue = "true")
public class McpSessionRegistry {

    private static final String KEY_PREFIX = "mcp:session:";

    @Value("${mcp.cluster.instance-address}")
    private String instanceAddress;   // 本实例的内网地址，如 http://10.27.233.18:8900/server

    @Value("${mcp.cluster.session-ttl:3600}")
    private long sessionTtlSeconds;

    // session 创建时注册：SET mcp:session:{id} → 本实例地址, EX 3600
    public void register(String sessionId) { ... }

    // 查询 session 归属实例：GET mcp:session:{id} → 实例地址
    public String resolve(String sessionId) { ... }

    // session 活跃时续期：EXPIRE mcp:session:{id} 3600
    public void renew(String sessionId) { ... }

    // session 断开时清理：DEL mcp:session:{id}
    public void remove(String sessionId) { ... }

    // 判断目标地址是否为本实例
    public boolean isLocal(String targetAddress) {
        return instanceAddress.equals(targetAddress);
    }
}

McpSessionRouteFilter.java — 核心过滤器 ​

@Order(0)，在 McpSignatureFilter(@Order(1)) 之前执行。

职责一：拦截 GET /sse — 即时注册

java
private void handleSseRequest(HttpServletRequest request, HttpServletResponse response,
                              FilterChain filterChain) throws ServletException, IOException {
    // 包装响应流，在 SSE 数据写出时实时捕获 sessionId
    SseSessionCaptureResponseWrapper wrapper =
        new SseSessionCaptureResponseWrapper(response, sessionRegistry);
    filterChain.doFilter(request, wrapper);
}

内部通过自定义 SessionCaptureOutputStream 代理原始 ServletOutputStream：

  • 每次 write() 时检查数据是否包含 sessionId=
  • 正则 sessionId=([a-f0-9\-]{36}) 提取 UUID
  • 匹配成功立即调用 sessionRegistry.register(sessionId)
  • 设置 volatile boolean registered = true，后续写入不再检查

职责二：拦截 POST /mcp/message — 路由转发

java
private void handleMessageRequest(...) {
    String sessionId = request.getParameter("sessionId");
    String targetAddress = sessionRegistry.resolve(sessionId);

    if (targetAddress == null) {
        // Redis 查不到 → 可能是本实例的 session（即时注册刚写入），放行本地处理
        filterChain.doFilter(request, response);
        return;
    }

    if (sessionRegistry.isLocal(targetAddress)) {
        // 是本实例 → 续期并放行
        sessionRegistry.renew(sessionId);
        filterChain.doFilter(request, response);
        return;
    }

    // 不是本实例 → HTTP 转发
    forwardRequest(request, response, targetAddress);
}

McpSessionSyncScheduler.java — 定时兜底 ​

java
@Scheduled(fixedDelay = 2000)  // 每 2 秒执行
public void syncSessions() {
    // 通过反射获取 WebMvcSseServerTransportProvider 内部 sessions 字段
    Map<String, ?> sessions = (Map<String, ?>) sessionsField.get(transportProvider);

    // 新增的 session → 注册到 Redis
    for (String sessionId : sessions.keySet()) {
        if (knownSessions.add(sessionId)) {
            sessionRegistry.register(sessionId);
        }
    }

    // 已断开的 session → 从 Redis 移除
    for (String sessionId : knownSessions) {
        if (!sessions.containsKey(sessionId)) {
            sessionRegistry.remove(sessionId);
            knownSessions.remove(sessionId);
        }
    }
}

5. 方案 C：Nginx sessionId 一致性哈希 ​

原理 ​

在 Nginx 层通过 URL 参数 sessionId 做一致性哈希，相同 sessionId 的请求始终路由到同一台实例。无需任何代码改动。

Nginx 配置 ​

nginx
upstream mcp_backend {
    # 根据 URL 中的 sessionId 参数做一致性哈希
    # 相同 sessionId 的请求始终路由到同一实例
    hash $arg_sessionId consistent;

    server 10.27.233.18:8900 weight=1;
    server 10.27.233.20:8900 weight=1;
    keepalive 64;
}

server {
    listen 443 ssl;
    server_name yjbackend.dajiaok.com;

    # MCP SSE + Message 端点
    location /server/sse {
        proxy_pass http://mcp_backend;

        # SSE 长连接必须的配置
        proxy_read_timeout 3600s;      # SSE 连接保持 1 小时
        proxy_buffering off;            # 关闭缓冲，SSE 实时推送
        proxy_cache off;
        proxy_http_version 1.1;
        proxy_set_header Connection '';  # SSE 不能用 keep-alive
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Host $host;
    }

    location /server/mcp/message {
        proxy_pass http://mcp_backend;

        proxy_read_timeout 60s;
        proxy_http_version 1.1;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Host $host;
    }
}

关键问题：SSE 请求没有 sessionId 参数 ​

GET /server/sse 是建立连接的请求，此时还没有 sessionId（sessionId 由服务端创建后通过 SSE 事件返回）。

hash $arg_sessionId 对 SSE 请求的值为空字符串，Nginx 会用空串做哈希 → 所有 SSE 连接请求都会路由到同一台实例。

解决思路：

nginx
# SSE 连接走 IP Hash（同一客户端的 SSE 固定到一台）
location /server/sse {
    proxy_pass http://mcp_backend_iphash;
}

# POST 消息走 sessionId Hash（按 session 路由）
location /server/mcp/message {
    proxy_pass http://mcp_backend_session;
}

upstream mcp_backend_iphash {
    ip_hash;
    server 10.27.233.18:8900;
    server 10.27.233.20:8900;
}

upstream mcp_backend_session {
    hash $arg_sessionId consistent;
    server 10.27.233.18:8900;
    server 10.27.233.20:8900;
}

前置条件 ​

  • 必须使用 Nginx 做负载均衡（替换阿里云 WAF 的负载均衡功能，或在 WAF 后面加一层 Nginx）
  • 所有 MCP 客户端 SSE 连接和 POST 消息使用相同的标识字段
  • 运维需要维护 Nginx upstream 列表（实例增减时需更新）

优劣 ​

  • ✅ 零代码改动
  • ✅ 保持 SSE 协议
  • ✅ 无转发延迟
  • ❌ 依赖 Nginx（当前架构是阿里云 WAF 直连，没有 Nginx 层）
  • ❌ SSE 请求没有 sessionId，哈希策略需要额外设计
  • ❌ 实例增减时需要更新 Nginx 配置
  • ❌ 无法在阿里云 WAF 层面实现（WAF 不支持自定义哈希规则）

适用场景 ​

适合已经使用 Nginx 做负载均衡、且不想改代码的团队。对于当前使用阿里云 WAF 的架构，需要在 WAF 和后端之间额外加一层 Nginx，引入了运维复杂度。

坑 2：instance-address 缺少 context-path 导致转发 404 ​

现象： [MCP-Cluster] Session 注册 日志正常，但客户端 POST 仍然 404。

原因： 服务配置了 server.servlet.context-path: /server。转发 URL 拼接逻辑为 targetAddress + request.getServletPath()：

  • http://10.x.x.x:8900 + /mcp/message = http://10.x.x.x:8900/mcp/message → 404（缺少 /server）

解决： MCP_INSTANCE_ADDRESS 必须包含 context-path：

# ❌ http://$(POD_IP):8900
# ✅ http://$(POD_IP):8900/server
1
2

坑 3：时间窗口 — 客户端 initialize 比定时同步快 ​

现象： 偶发 "Session not found"，但稍后重试又正常。

原因： SSE 连接建立后客户端立即发 initialize POST。定时扫描器每 2 秒执行一次，在这 2 秒内 sessionId 尚未同步到 Redis，跨实例路由失败。

解决： 在 McpSessionRouteFilter 中包装 SSE 响应的 ServletOutputStream，通过正则 sessionId=([a-f0-9\-]{36}) 实时提取 sessionId 并立即注册到 Redis，消除时间窗口。

坑 4：环境变量占位符无默认值导致启动失败 ​

现象： Could not resolve placeholder 'MCP_INSTANCE_ADDRESS' in value "${MCP_INSTANCE_ADDRESS}"

原因： Nacos 配置中使用了 ${MCP_INSTANCE_ADDRESS} 但本地开发/环境变量未注入时 Spring 无法解析。

解决： 加默认值：${MCP_INSTANCE_ADDRESS:http://localhost:8900/server}

7. 验证方式 ​

部署后搜索日志关键词 [MCP-Cluster]：

日志含义异常排查
SessionSyncScheduler 初始化成功三个核心 Bean 加载成功没有此日志 → 检查 Nacos mcp.cluster 是否顶层 key
SSE 即时注册 sessionId=xxx即时注册生效没有此日志 → Filter 可能被跳过，检查 @Order
Session 注册: sessionId=xxx, instance=http://10.x.x.x:8900/serverRedis 注册成功instance 地址不含 /server → 改环境变量
Session 路由转发: sessionId=xxx, target=...跨实例转发触发转发后 404 → 检查 target 地址是否正确

如果日志中完全没有 [MCP-Cluster] 输出 → Bean 未加载，99% 是配置路径问题。

8. 决策结论 ​

最终采用方案 B（SSE + 应用层路由转发）。

决策因素说明
客户端零改动已有多个客户端接入，改配置成本高
保留变更通知SSE 支持服务端主动推送 tools/resources 变更
网关无需变动不需要运维介入
方案 A 备用已验证可行，未来如需无状态可随时切换
方案 C 不适用当前无 Nginx 层，且 WAF 不支持自定义哈希
