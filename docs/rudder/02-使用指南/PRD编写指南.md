# PRD 编写指南

PRD（Product Requirements Document）是 Phase 1 的产出物，也是 AI 写代码的依据。

## PRD 在哪

```
.rudder/tasks/MM-DD-任务名/prd.md
```

## PRD 由谁写

你和 AI 一起写。流程：

1. AI 问你几个问题，了解需求背景
2. AI 把结论写入 `prd.md`
3. 你确认或修改
4. 反复 1-3 直到 PRD 完整

## 好的 PRD 长什么样

### 反例（太模糊）

```markdown
# 用户登录

实现一个登录功能。
```

这等于没写。AI 不知道用什么认证、有没有注册、需不需要记住登录状态。

### 正例（具体）

```markdown
# 用户登录

## 目标
实现基于 JWT 的用户登录系统，支持用户名+密码登录。

## 需求
- 登录接口：POST /api/auth/login
- 返回 JWT token，有效期 7 天
- 登录失败返回 401 + 错误原因
- 同一 IP 每分钟最多 5 次登录尝试
- 密码使用 bcrypt 加密，不存明文

## 技术选型
- JWT: java-jwt 4.x
- 密码加密: BCryptPasswordEncoder（Spring Security）
- 存储: HttpSession 不用了，纯无状态 JWT

## 前端
- React + Ant Design Form
- 表单校验：用户名必填、密码 6-20 位
- 登录成功后 token 存入 localStorage

## 非功能性要求
- 登录接口 P99 < 200ms
- 支持 CORS
```

## PRD 应该包含什么

| 章节 | 是否必填 | 说明 |
|------|---------|------|
| 目标 | ✅ 必填 | 一句话描述做什么 |
| 需求 | ✅ 必填 | 具体的功能点、接口、交互 |
| 技术选型 | 复杂任务必填 | 用什么库/框架，版本 |
| 前端需求 | 有前端时必填 | UI 组件、交互、状态管理 |
| 非功能性要求 | 推荐 | 性能、安全、兼容性 |
| 研究结论 | 可选 | 调研产出的技术决策 |

## 什么时候更新 PRD

以下情况需要更新 PRD：

| 场景 | 说明 |
|------|------|
| 需求变更 | 用户改了主意 |
| 技术调研后发现方案不可行 | 换了技术方案 |
| 开发中发现遗漏的场景 | 补上遗漏的边界条件 |
| 安全评审后新增要求 | 加了权限控制 |

AI 在开发过程中如果发现需求不明确，会暂停开发，先和你确认需求，然后更新 PRD。

## PRD 和工作流的关系

```
PRD 不完整 → AI 会问你问题 → 补充 PRD → 你确认 → 进入实现
```

AI 不会在 PRD 模糊的情况下盲目写代码。这是 Rudder 和直接聊天的区别——有 PRD 这个"关卡"挡在前面。
