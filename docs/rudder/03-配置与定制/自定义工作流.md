# 自定义工作流

每个团队的工作方式不同。Rudder 的核心流程是固定的，但可以在多个层面定制。

## 定制编码规范（Spec）

最常见的定制方式。Rudder 的 Spec 就是你项目的编码标准。

### 添加自定义 Spec

在 `.rudder/spec/` 下创建新文件：

```
.rudder/spec/
└── backend/
    ├── index.md
    └── my-custom-rule.md    # ← 你的自定义规范
```

然后在 `index.md` 中添加链接：

```markdown
- [自定义规则](./my-custom-rule.md)
```

AI 在相关场景下会自动读取这个文件。

### 删除不适用的 Spec

初始化后的 Spec 是通用模板。不需要的部分直接删除：

```bash
# 如果你没有前端
rm -rf .rudder/spec/frontend/
```

### 修改 Spec 内容

直接编辑 `.rudder/spec/` 下的文件。建议：

- 用具体代码示例，不要写空话
- 标注"为什么"，不只是"怎么做"
- 指向真实的源码文件

## 定制工作流阶段

`.rudder/workflow.md` 定义了 4 阶段循环。你可以：

### 添加自定义阶段

在 `workflow.md` 中添加新的 `[workflow-state:xxx]` 块：

```markdown
[workflow-state:in_review]
当前任务进入代码审查阶段。
AI 应该：1. 创建 PR  2. 指派 reviewer  3. 等待审查通过
[/workflow-state:in_review]
```

状态名规则：`[A-Za-z0-9_-]+`

### 修改现有阶段

直接编辑 `workflow.md` 中各阶段的内容。改完后运行：

```bash
rudder update
```

这会把新配置推送到各平台的 hooks 中。

### 添加自定义状态到任务

修改 `task.json` 的 `status` 字段，或通过钩子自动修改：

```json
{
  "hooks": {
    "after_archive": ["python3 my_scripts/notify_team.py"]
  }
}
```

## 定制生命周期钩子

### 在 task.json 中声明

```json
{
  "id": "my-task",
  "title": "我的任务",
  "hooks": {
    "after_create": [
      "echo 新任务已创建"
    ],
    "after_archive": [
      "python3 scripts/notify_slack.py"
    ]
  }
}
```

### 支持的事件

| 事件 | 触发时机 |
|------|---------|
| `after_create` | 任务创建后 |
| `after_start` | 任务激活后 |
| `after_finish` | 任务清理指针后（不等于归档） |
| `after_archive` | 任务归档后 |

## 定制 config.yaml

```yaml
# 调整日志行数
max_journal_lines: 5000

# 开启自动提交（谨慎开启）
session_auto_commit: true

# 代码自动提交（跳过确认，不推荐）
code_auto_commit: true

# 修改提交消息模板
session_commit_message: "chore({{task}}): {{summary}}"
```

## 定制平台配置

如果你想在某个平台添加自定义行为，直接编辑平台目录下的文件：

```
.claude/
├── settings.json           # 自定义 Claude Code 设置
├── commands/               # 自定义 slash 命令
└── agents/                 # 自定义 agent

.cursor/
├── rules.md                # 自定义 Cursor 规则
└── hooks/                  # 自定义 hooks
```

> 注意：`rudder update` 不会覆盖你自己修改的平台文件。

## 定制建议

1. **Spec 大胆改**——这是你的编码标准，越具体越好
2. **workflow.md 谨慎改**——改动后确保各平台 hooks 能正确解析
3. **平台配置自由改**——这是你自己的项目，Rudder 不会覆盖
4. **钩子逐步加**——先加简单的 `echo`，确认能跑再加复杂脚本
