# 日常使用指南

这一章不讲原理，只讲**日常开发中怎么用 Rudder**。

## 场景一：开发新功能

```
用户："加个用户登录功能"
```

### 1. AI 自动创建任务

AI 会先问你几个问题确认需求，然后自动创建任务：

```bash
python3 .rudder/scripts/task.py create "用户登录功能" --slug user-login
```

创建后进入 **Phase 1: Plan**，AI 会：
- 和你讨论需求细节
- 把结论写入 `prd.md`
- 调研需要用到的库和方案
- 填充 `implement.jsonl`（告诉实现代理需要读哪些 Spec）

### 2. 激活任务，开始写代码

PRD 完成后，你确认没问题，AI 激活任务：

```bash
python3 .rudder/scripts/task.py start user-login
```

状态变为 `in_progress`，进入 **Phase 2: Execute**。AI 会派子代理写代码。

### 3. 代码审查

写完代码后，AI 派另一个子代理审查（`rudder-check`），对照 Spec 逐项检查：
- 编码规范是否遵守
- lint、type-check 是否通过
- 有没有潜在问题

### 4. 沉淀认知，提交代码

审查通过后，进入 **Phase 3: Finish**：
- 把本轮学到的新规范写回 Spec（下次就不会忘了）
- AI 展示变更文件，生成提交计划
- 你确认后，AI 执行 `git commit`
- 提醒你运行 `/finish-work` 收尾

## 场景二：修 Bug

```
用户："登录接口 500 了，查一下"
```

### 简化流程

修 Bug 通常需求很明确，不需要长 PRD：

1. AI 创建任务（带简短描述）
2. 直接进入实现阶段
3. 修复后审查
4. 提交代码

如果 Bug 比较复杂，AI 还是会走完整流程，但 PRD 会很短。

## 场景三：代码重构

```
用户："把用户模块的 Controller 拆分成 Service 层"
```

重构的特点是：
- PRD 主要是技术方案（改哪些文件、怎么改）
- 审查重点是不改行为
- Spec 更新可能比较大（如果重构改变了编码规范）

## 场景四：简单修改（跳过流程）

```
用户："把这个变量名改一下"
```

特别小的修改，你可以说以下任一话术跳过任务流程：

- "skip trellis" / "别走流程"
- "直接改" / "小修一下"
- "先别建任务"

AI 看到这些话就会直接改，不创建任务。

## 日常开发节奏

```
早上：  新需求 → Phase 1 讨论需求 → 确认 PRD
上午：  Phase 2 写代码 → Phase 3 审查 + 提交
下午：  新需求 → 重复
下班前：/finish-work 归档任务，记录工作日志
```

## 常用操作速查

| 你想做什么 | 对 AI 说 |
|-----------|---------|
| 开新需求 | "我想做 XXX" |
| 修 Bug | "XXX 出了个问题" |
| 跳过流程 | "skip trellis" / "直接改" |
| 查看当前任务 | "当前在做什么任务？" |
| 查看工作日志 | "我上次做了什么？" |
| 更新 Spec | "这条规则应该写进 Spec" |
| 结束今天工作 | "/finish-work" |

## 工具路径管理

第一次遇到工具找不到的情况时：

1. AI 会在 session 中显示 `<tool-paths>` 块，列出所有工具和当前路径
2. 如果路径不对，你可以：
   - 告诉 AI 正确路径
   - 或者直接编辑 `.rudder/config_local.yml`
3. AI 会自动把发现的工具追加到 `config_local.yml`

> 小贴士：`config_local.yml` 是 git 忽略的，不会被别人覆盖。
